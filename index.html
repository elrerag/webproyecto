PROYECTO FINAL CURSO JAVA FULL STACK ☕️

Objetivo del proyecto.

Diseñar una solución tecnológica MVP (Producto mínimo viable), utilizando un sitio web. El alcance estará definido por el contenido visto en el curso. En el caso que el alcance del proyecto exceda los recursos disponibles (1 desarrollador y 2 semanas para la entrega), se debe limitar a un producto que apunte al destino de la aplicación, dejando este, preparado para escalar las nuevas características en una nueva iteración posterior al curso.

Requerimientos técnicos (ligeramente flexible 🧠):

  • Lenguaje de programación:
    • Java.
    • Javascript
  • Lenguajes estructurales:
    • HTML.
    • XML.
    • CSS.
  • Frameworks:
    • Spring (2.4.0-SNAPSHOT) MVC, con dependencias controladas por MAVEN.
    • BOOTSTRAP 4
  • Librerías (opcionales, si llegan a ser necesarias):
    • Algunas tentativas:
      • JQuery
      • https://www.chartjs.org/
      • https://datatables.net/
      • Otras propuestas son aceptables, previa evaluación.
  • Lenguajes para la vista.
    • JSP
    • JSTL
  • Motor de base de datos (Cualquier opción).
    • SQLITE
    • MYSQL
    • ORACLE 🤢
  • IDEs
    • Aplicación principal
      • STS 4
    • Apoyo para estructurar vistas si se requiere.
      • VSCode
      • Atom
      • Sublime
  • Versionamiento
    • GIT
    • GitHub

Hitos Entregadles

Dividiremos el trabajo en cuatro etapas principales, las cuales serán medidos y evaluados según completitud y calidad.

Primer hito 🎱

  • Fecha de inicio: Viernes 24 de Julio.
  • Fecha de término: Martes 28 de Julio.
  • Días totales: 4
  • Objetivo principal:
    • Concepción del proyecto y planificación del desarrollo.
      • Idear un proyecto que cumpla con las exigencias necesarias para su evaluación.
        • Iteraciones tentativas luego de tener la idea.
          • Evaluar el mercado de aplicaciones, que resuelvan el mismo problema o uno similar, sacando ideas de la interacción de usuario además de interfaces gráficas, no es necesario que el proyecto no exista, podemos crear uno que se parezca mucho a alguno existente. Lo más probable es que exista un producto similar, lo que es bastante normal en la industria. En este caso necesitamos una característica que lo destaque que no sea necesariamente nueva, pero personalizada.
          • Teniendo ya el punto anterior, es hora de categorizar a los usuarios (arquetipos). Debemos definir quienes serán las personas que utilizarán el sistema, cuál es el ROL 👀, de cada uno y qué servicios requieren utilizar del sistema. (Requerimientos - propuesta de diagrama de casos de uso de los servicios con las historías de usuario)
          • El siguiente paso se basa en desarrollar un mapa del flujo del sistema, con él, podemos acercarnos a la estimación del alcance total, además de evaluar posibles cambios de la idea original. El diagrama nos expone como cada vista se relaciona con otra y por qué motivo (el contenido de las vistas no se detalla, mantener el diagrama lo más simple posible) Ejemplo:
          • Teniendo el flujo de nuestra aplicación, podemos obtener una estimación de las entidades necesarias, además de la obtención del modelo de datos que lo soportará.
          • Finalmente podemos bosquejamos nuestros mockups o prototipos de interfaz a bajo nivel, que expresen el contenido y posible estructura de cada una de nuestras vistas.
          • Iterar los puntos que sea necesarios.
    • Entregables a evaluar
      • Artefactos propuestos (Herramientas de diseño a libre elección 🎨).
        • El resultado de la evaluación, dependerá del aporte real que proporciones los artefactos a la implementación del sistema.

Segundo hito 🎱

  • Fecha de inicio: Miércoles 29 de Julio.
  • Fecha de término: Viernes 31 de Julio. (Fecha de entrega hasta domingo 02 de Agosto)
  • Días totales: 3 Clases + 2 Fin de semana (opcional).
  • Objetivo principal:
    • Con nuestros mockups, podemos tener una idea muy cercana a las vistas que necesitemos. El objetivo es preparar las vistas que soportarán el sistema.
    • Para cumplir con el objetivo podemos usar alguna plantilla (html, css (Bootstrap 4), js), además de poder desarrollarla completamente. Luego la adaptamos a nuestros mockups, para finalmente pasar estas vistas con JSP y JSTL.
    • Debemos tener en cuenta que las vistas tendrán, controladores que permitan interactuar con la presentación con datos simulados, es decir que los controladores, solo accedan a la vista y no se vaya a un servicio para la obtención de los datos, si no que estos pueden ser simulados si es que el desarrollo de esta simulación es trivial, en caso contrario, si presentar datos simulados requiere una alta carga de desarrollo, entonces prescindir de la simulación, limitándose a mostrar la vista correspondiente, dependiendo de la URI solicitada.
  • Entregables a evaluar:
    • Proyecto Spring, el cual sea capaz de mostrar las vistas según URL, estas vistas, deberán ser de alta fidelidad, es decir que mientras menos cambios requieran en las siguientes etapas, más alta será la evaluación.
    • El proyecto deberá estar versionado, en un repositorio GitHub, además de subirlo a la plataforma LMS.
    • Las dependencias (maven) que deberá incluir el proyecto, dependerá de una evaluación previa a cada desarrollo.
